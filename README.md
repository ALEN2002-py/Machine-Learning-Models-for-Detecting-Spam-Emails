## Dataset

The [UCI Spambase dataset](https://archive.ics.uci.edu/dataset/94/spambase) contains **4,601 emails** (1,813 spam / 2,788 legitimate) described by **57 continuous features**:

| Feature group | Count | Description |
|---|---|---|
| `word_freq_*` | 48 | Percentage of words matching a given keyword |
| `char_freq_*` | 6 | Percentage of characters matching special chars (`;`, `(`, `[`, `!`, `$`, `#`) |
| `capital_run_length_*` | 3 | Statistics on runs of consecutive capital letters (average, longest, total) |

Target column: `class` вЂ” `1` (spam) or `0` (legitimate).

The dataset is **pre-included** at `data/spambase.csv`. It is also available at the [UCI ML Repository](https://archive.ics.uci.edu/dataset/94/spambase).

## Quickstart

### 1. Clone & install

```bash
git clone https://github.com/ALEN2002-py/spam-email-detector.git
cd spam-email-detector
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### 2. Train all models

```bash
python src/train.py
# Optional flags:
# --data data/spambase.csv   (default)
# --output results/          (default)
```

This will:
- Load and scale the dataset with a stratified 80/20 split
- Run 5-fold cross-validation for each sklearn model
- Train Random Forest, Logistic Regression, SVM, and the ANN
- Save all models to `models/`
- Write all plots and a `metrics.json` to `results/`
- Print a final leaderboard to stdout

### 3. Explore with EDA

```bash
python src/eda.py
```

### 4. Run inference

```bash
# Single email (provide 57 feature values)
python src/predict.py --model random_forest --features 0,0.64,0.64,...

# Batch prediction from CSV
python src/predict.py --model svm --csv path/to/new_emails.csv
```

### 5. Run tests

```bash
pip install -r requirements-dev.txt
pytest tests/ -v --cov=src
```

## Models & Design Decisions

### Random Forest
- **n_estimators=300**, criterion=entropy, max_depth=15
- Ensemble of decision trees вЂ” naturally handles non-linear feature interactions and the high-dimensional word-frequency space
- Provides Gini-based feature importances at no extra cost

### Logistic Regression
- LBFGS solver, max_iter=5000
- Strong baseline; interpretable coefficients map directly to feature contributions
- Particularly effective here because many features are near-linearly separable after scaling

### SVM (RBF kernel)
- `probability=True` enables ROC-AUC computation via Platt scaling
- `gamma='scale'` (= 1 / (n_features Г— X.var())) вЂ” data-adaptive bandwidth
- Maximises the margin between spam and legitimate hyperplanes

### ANN
- Architecture: Input(57) в†’ Dense(128, ReLU) в†’ BN в†’ Dropout(0.3) в†’ Dense(64, ReLU) в†’ BN в†’ Dropout(0.2) в†’ Dense(32, ReLU) в†’ Dense(1, sigmoid)
- Trained with Adam (lr=1e-3), binary cross-entropy, batch size 32
- **EarlyStopping** (patience=15) and **ReduceLROnPlateau** prevent overfitting
- BatchNormalization stabilises training on the heterogeneous spambase feature scale

### Preprocessing
- **StandardScaler** (zero mean, unit variance) is applied to all models вЂ” not just the scale-sensitive ones (LR, SVM, ANN). This ensures a fair comparison and speeds up tree-based model convergence
- Scaler is **fit only on training data** and serialised with the model artefacts to prevent data leakage during inference

## Key Findings

1. **Random Forest is the top performer**, achieving ~95.3% accuracy and ~98.7% AUC вЂ” consistent with the literature on tree ensembles for tabular spam classification.
2. **Capital-run-length features** (`capital_run_length_total`, `capital_run_length_longest`) are among the strongest spam signals вЂ” emails with long unbroken caps runs are disproportionately spam.
3. **High-stakes keywords** (`word_freq_free`, `word_freq_money`, `word_freq_000`) show the highest univariate correlation with the spam label.
4. **Logistic Regression** underperforms relative to the other models on this dataset (non-linear interactions matter) but remains a useful baseline for interpretability.
5. **ANN** matches SVM but requires substantially more training time, suggesting a complexity-accuracy trade-off worth considering in production settings.
